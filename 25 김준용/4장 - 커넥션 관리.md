### TCP 스트림이 데이터를 전송하는 방식
- TCP는 IP 패킷(or IP 데이터그램)이라 불리는 작은 데이터 조각을 통해 데이터를 전송한다.

- HTTP가 메시지를 전송하여 하면 TCP는 세그먼트라는 단위로 데이터 스트림을 잘게 나누고, 이를 IP 패킷에 담아서 전송한다.

IP 패킷은 크게 3가지로 나뉜다.
  1) IP 패킷 헤더(~20 bytes): 목젖지 IP주소, 크기, 기타 플래그
  2) TCP 세그먼트 헤더(~20 bytes): TCP 포트 번호, TCP 제어 플래그, 데이터 순서 & 무결성 검사용 숫자 값.
  3) TCP 데이터 조각(0>= bytes)

## TCP의 성능에 대한 고려
HTTP는 TCP 바로 위의 계층이므로 TCP의 성능에 되게 민감하다. 그러므로 HTTP의 성능을 최대한 뽑을려면, TCP에 대해 잘 알아야 한다.

### HTTP 트랜잭션 지연
데부분 HTTP 지연은 TCP 네트워크 지연 때문에 발생한다.

  1) IP 주소 & 포트: 방문한적 없는 호스트는 DNS 이름 분석을 사용하는데, 이는 수십 초의 시간이 걸릴수 있다.
  2) TCP 연결 요청 & 허가: 1~2초만 걸리지만 수백개의 HTTP 트랜잭션이 만들어지면 빠르게 쌓인다
 3) HTTP 요청 처리 시간: 메시지 전달, 처리 응답에 시간 소요
  4) 하드웨어, 전송 속도, 메시지 크기, 물리적 거리: 굳이 설명???

### TCP 커넥션 핸드셰이크 지연
전송할 데이터가 작으면 작을수록 HTTP 선능은 저하 시킬수 있다.

TCP 핸드셰이크 순서를 통해 이해:
  1) 서버에 'SYN'이라는 커넥션 생성 요청 플래그를 가진 TCP 패킷(40~60 bytes) 보낸다
  2) 서버는 몇 가지 커넥션 매개변수를 산출하고, '요청OK'를 의미하는 'SYN'과 'ACK'플래그를 포함한 TCP 패킷을 보낸다.
  3) 연결 OK를 알리기 위해 서버에 다시 확인응답 신호를 보낸다. (요즘 TCP는 이거에 데이터도 보낼수 있음)

이는 데이터 크기가 작을수록 50% 이상의 HTTP 트랜잭션 시간을 쓸수 있다. (아니 상태 코드 보낸다고 이짓을...)

### 확인응답 지연
성공적 데이터 전송은 프로토콜이 담당한다.
> 이게 뭔 당연한 소리?

> 라우터는 과부화 걸리면 패킷이 먼저 죽는다. TCP같은 프로토콜이 데이터의 온전함을 담당해야 함. ( Your data is not the internet's priority (: ) 

> 그럼 어케?

> 데이터 무결성 체크섬.

각 세그먼트 수신자는 세그먼트를 온전히 받으면 작은 확인 응답 패킷을 송신자로 보낸다. 이를 통해 패킷에 문제가 생기면 다시 보낼수 있다.

심지어 이 패킷은 은근 작아서, 같은 방향으로 송출되는 데이터 패킷에 편승 시킨다.

효율성 증가를 위하여 0.1~0.2초 동안 확인응답을 버퍼에 넣고, 편승 시키기 위한 데이터 패킷을 찾는 알고리즘 도입도 하였다.

**하지만** HTTP의 요청&응답 동작 방식은 이 '편승' 기회를 많이 감소시킨다. 막상 하려고 하면, '편승'할 패킷을 거의 못찾기 때문에, 오히려 *지연할때가 많다*.

Q: 바꾸면 되지 않나?

A: 가능한데, 모르면 건들지 마셈. TCP는 인터넷을 개인의 무식함으로서 막아주는 알고리즘이 많은데, 그거 잘못 건들면 망할수 있음.

### TCP 느린 시작
slow start: TCP가 인터넷의 급작스러운 부하와 혼잡을 방지하는 데 쓰는 도구.

what is it: 성공적인 요청&응답 을 기준으로 패킷 전송 총량 증가 알고리즘

알고리즘: (S = 0, S+=1 if ACK, packet_transfer_capacity=2**S) 

이 과정의 이름: '혼잡 윈도를 연다(opening the congestion window)' (이름이 참 창의적이군)

### 네이글(Nagle) 알고리즘 & TCP_NODELAY
네이글(Nagle) 알고리즘: 작은 크기의 데이터를 포함하는 많은 수의 패킷 전송으로 인한 네트워크 성능의 저하를 감소 시키는 알고리즘.

알고리즘:

  if there is new data to send then

    if the window size ≥ MSS and available data is ≥ MSS then
        send complete MSS segment now
    else
        if there is unconfirmed data still in the pipe then
            enqueue data in the buffer until an acknowledge is received
        else
            send data immediately
        end if
    end if
  end if

**요기 적힌 알고리즘은 완전히 증명되지는 않았습니다!** 

현실: 어지간하면 형편없음. 추가 패킷 기다리는게 지연 증가.

> For more details: RFC896 "Congestion Control in IP/TCP Internetworks"

### 흔히 잘못 이해하는 Connection 헤더
커넥션 토큰을 쉼표로 구분하여 가지고 있으며, 그 값들은 다른 커넥션에 전달되지 않는다.
홉별(hop-by-hop) 헤더 명을 기술 >> '헤더 보호하기'라 부름.

보낼수 있는 토큰 종류:
1) HTTP 헤더 필드 명은, 이 커넥션에만 해당되는 헤더들을 나열한다.
2) 임시적인 토큰 값은, 커넥션에 대한 비표준 옵션을 의미한다
3) close 값은, 케넥션이 작업이 완료되면 종료되어야 함은 의미한다.

### 순차적인 트랜잭션 처리에 의한 지연
각 트랜잭션이 새로운 커넥션이 필요로 한다면, 커넥션을 맺는데 발생하는 지연과 함께 느린 시작 지연이 발생할 것이다.

Potential Solutions:
  1) 병렬(parallel) 커넥션: 여러 개의 TCP 커넥션을 통한 동시 HTTP 요청
   2) 지곳(persistent) 커넥션: TCP 커넥션 재활용
  3) 파이프라인(piplined) 커넥션: 공유 TCP 커넥션을 통한 병렬 HTTP 요청
  4) 다중(multiplexed) 커넥션: 요청과 응답들에 대한 중재[EXPERIMENTAL]

## 지속 커넥션
연결한 서버의 TCP커넥션을 유지하여 재사용하는 방식.
Why: 느린 시작 피하기, TIME_WAIT 피하기, 시간&자원 절약(어차피 같은 서버 리소스 사용일 가능성이 높기 때문)

### 지속 vs 병렬 커넥션
상황이 맞으면, 병렬 커넥션은 엄청나다. 하지만 여러 단점이 있다.
- 각 트랜잭션마다 새로운 커넥션을 맺고 끊기 때문에 시간이 대폭 소요
- TCP 느린 시작에 자주 걸림 (위의 이유로)
- 실제 연결 양은 제안적 (주로 4개)

이에 비해 지속 커넥션은 몇 가지 장점이 있다
- 커넥션을 맺기 위한 사전 작업과 지연 감소
- TCP 느린 시작 피함
- 커넥션 수 줄이기
그러나 단점도 있다.
- 커넥션 Stack
- No 자동 close

### Keep-Alive 동작
keep-alive 는 HTTP/1.1 명세에서 빠졌지만, 아직 널리 사용되고 있기 때문에, 이것을 고려하고 HTTP  애플리케이션을 개발해야 한다. (자세한 설명은 RFC 2068 과 같은 명세 참고)

- HTTP/1.0에서 커넥션을 유지(keep-alive)하려면, 클라이언트가 요청에 `Connection: keep-alive` 헤더를 포함시켜야 한다.  

- 서버가 이후 요청도 동일한 커넥션을 통해 받기를 원한다면, 응답 메시지에도 동일한 `Connection: keep-alive` 헤더를 포함시켜야 한다.  
- 만약 응답에 해당 헤더가 포함되어 있지 않다면, 서버가 keep-alive를 지원하지 않거나 커넥션을 유지할 의사가 없다는 뜻이며, 응답 전송 후 커넥션을 종료할 것으로 추정된다.

### Keep-Alive 옵션
keep-alive는 요청이자 옵션이다. 서버나 클라이언트는 무조건 요청을 따를 필요는 없다.
*보내는대로 동작한다는 보장은 없다*

Keep-Alive 헤더는 선택 사항이지만, Connection: Keep-Alive 헤더가 있을 때만 사용할수 있다

### Keep-Alive 커넥션 제한과 규칙
- HTTP/1.0에서 기본으로 사용되지 않는다. 쓰고 싶으면, Connection: Keep-Alive 요청 헤더를 보내야 한다.
- 커넥션을 계속 유지하려면 모든 메시지에 Connection: Keep-Alive 헤더를 포함해야 한다. 
> 사실 잘 모르겠음, 구글은 선택적이라는데?
- 클라이언트는 Connection: Keep-Alive 응답 헤더가 없으면, 서버가 응답 후에 커넥션을 끊을 것임을 알 수 있다.
- 커넥션이 끊어지기 전에 엔터티 본문의 길이를 알 수 있어래 커넥션을 유지할 수 있다. 정확한 Content-Length 값과 함께, 멀티파트 미디어 형식을 가지거나, 청크 전송 인코딩으로 이코드 되어야 함을 뜻한다. Content-length 값을 잘못 보내면 기존의 메시지의 끝과 시작을 정확히 알수 없음을 주의해라.
- 프락시와 게이트웨이는 반드시 메시지를 전달하거나 캐시에 넣기 전에 Connection 헤더에 명시된 모든 필드와 Connection 헤더를 제거해야 한다.
- keep-alive 커넥션을 인식 못하는 서버와 keep-alive 커넥션을 맺일려고 하지 마라. Dumb 프락시로 인해 고생한다.
-  HTTP/1.0을 따르는 기기로부터 받은 모든 Connection 헤더 필드는 무시해야 한다. 안그러면 행(hand)이 걸릴수 있는 위험이 있다.
- **클라이언트는**, 응답 전체를 모두 받기 전에 커넥션이 끊어졌을 경루, 별다른 문제가 없으면 요청을 다시 보낼 수 있게 준비되어 있어야 한다.

2) 서버는 Connection: Keep-Alive를 보내고, 연결을 프락시와 유지한다.
3) 프락시가 받은 메시지를 클라이언트로 보내고, 커넥션 close를 기다린다. 서버는 keep-alive를 간주하고 연결을 유지한다.
4) 메시지를 받은 크라이언트는 프락시로 요청을 보낸다.
5) 프락시는 서버의 연결이 끊어지기를 대기하고 있어서, 클라이언트의 메시지를 무시한다.
6) Timed Out.

### Proxy-Connection 
멍청한 프락시의 문제를 해결하기 위해 제시된 차선책.
How: Proxy-Connection 헤더를 받은 서버는 이를 무시하고, 영리한 프락시는 이를 받으면, 'Connection: Keep-Alive'으로 바꿔 웹 서버에 전송하게 만든다.

단점: 클라이언트와 서버 사이의 프락시는 하나가 아닐때가 많다. 멍청한 프락시와 영리한 프락시가 섞여 있을 가능성이 있으며, 네트워크상 멍청한 프락시가 '보이지 않는'경우가 많다.

### HTTP/1.1의 지속 커넥션
keep-alive 커넥션 대신 설계가 거 된 지석 커넥션 지원.
- 기본 활성(닫을려면, Connection: close 헤더 명시)
-  Connection: close 이 없어도 클라이언트나 서버측에서 끊을수 있음

### 지속 커넥션의 제한과 규칙
- 클라이언트가 요청에 Connection: close 헤더를 포함해 보냈으면, 클라이언트는 그 커넥션으로 추가적인 요청을 보낼 수 없다.
- 클라이언트가 해당 커넥션으로 추가적인 요청을 보내지 않을 것이라면, 마지막 요청에 Connection: close 헤더를 보내야 한다.
- 커넥션에 있는 모든 메시지가 자신의 길이 정보를 정확히 가지고 있을 때에만 커넥션을 지속시킬 수 있다.
- HTTP/1.1 프락시는 클라이언트와 서버 각각에 대해 별도의 지속 커넥션을 맺고 관리해야 한다.
- HTTP/1.1 프락시 서버는 클라이언트가 커넥션 관련 기능에 대한 클라이언트의 지원 범위를 알고 있지 않은 한 지속 커넥션을 맺으면 안 된다.
- HTTP/1.1 기기는 Connection 헤더의 값과 상관없이 언제든지 커넥션을 끊을 수 있다
- HTTP/1.1 애플리케이션 중간에 끊어지는 커넥션을 복구할 수 있어야만 한다.
- 클라이언트는 전체 응답을 받기 전에 커넥션이 끊어지면, 요청을 반복해서 보내도 문제가 없는 경우에는 요청을 다시 보낼 준비가 되어 있어야 한다.
- 하나의 클라이언트는 넉넉잡아 주 개의 지속 커넥션만을 유지해야 한다.

## 파이프라인 커넥션
빠르지만 제약이 많은 커넥션

- 클라이언트는 지속 커넥션인지 확인하기 전까지는 파이프라인을 이어서는 안 된다.
- HTTP 응답은 요청 순서와 같이 와야 한다.
- HTTP 클라이언트는 커넥션이 언제 끊어지더라도, 완료되지 않은 요청이 파이프라인에 있으면 언제든 다시 요청을 보낼 준비가 되어 있어야 한다.
- 클라이언트는 반복해서 문제가 생기는 요청은 파이프라인을 통해 보내면 안 된다.

## 커넥션 끊기에 대한 미스터리
### '마음대로'
HTTP 애플리케이션은 언제든지 TCP커넥션을 끊을 수 있다. 특히 에러가 난 상황에서 갑자기 끊길수 있다.

### Content-Length & Truncation
일부 오래된 서버는 Content-Length를 커넥션 종료때 생략하기 때문에 서버한테 다시 물어봐도 답이 오지 않을수 있다는 것을 고려하여 개발해야 한다.

## 우아한 커넥션 끊기
단순한 애플리케이션이 않는 경우, '절반 끊기'를 사용해야 한다.
자신의 출력 태널을 먼저 끊고 다른 쪽에 있는 기기의 출력 채널이 끊기는 것은 기다리는 것이다.
양쪽에서 더는 데이터를 전송하지 않을 것이라고 알려주면, 위험 없이 온저히 종료된다.
